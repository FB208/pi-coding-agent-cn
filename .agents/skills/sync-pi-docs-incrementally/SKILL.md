# 增量同步 Pi 中文文档

## 目标

以 `content/source.json` 中的上游提交为唯一基线，对比 `earendil-works/pi` 的目标提交。只处理比较结果中发生变化的文档、导航、图片和包版本，原样保留未变化的中文章节。

执行前完整阅读 [增量更新契约](references/update-contract.md)。契约与本文件冲突时，以契约中更严格的停止条件为准。

## 执行流程

1. 在仓库根目录检查 `git status --short`，辨认并保留用户已有改动。不得覆盖或清理任务范围外的文件。
2. 运行只读计划：

   ```powershell
   node scripts/sync-upstream.mjs --plan
   ```

3. 检查计划中的基线提交、目标提交、版本变化和相关文件，保存计划返回的完整目标提交 SHA。即使没有上游变化，也继续执行下一步完成站点验收和构建。
4. 使用固定 SHA 执行同步，禁止改用浮动分支：

   ```powershell
   node scripts/sync-upstream.mjs --target <目标提交SHA>
   ```

   该命令会在同一事务中完成结构检查、增量测试、`npm run build:pages`、`npm run build` 和完整测试。任一步出现可捕获异常或命令非零退出时，自动恢复文档、图片、导航、结构、日志、基线与生成数据；进程被强制终止不在此保证内。不得在命令后运行会全量改写 Markdown 的规范化脚本。
5. 检查最终差异。确认未出现在计划中的中文正文没有变化，`content/changelog.json` 至多新增一个对应目标提交的条目，且没有生成 Markdown 格式的变更日志。

## 必须遵守

- 只把 GitHub Compare API 返回的相关文件作为更新输入，不枚举或下载全部上游文档来寻找变化。
- 所有上游内容都从同一个不可变目标提交 SHA 获取，包括版本、导航、Markdown 和图片。
- 修改页面按标题章节合并。内容完全相同的章节必须逐字节复用现有中文；移动但未修改的章节也必须复用。
- 新增章节和内容变化的章节才允许调用翻译模型。新增页面可以整页翻译；禁止因单章变化重译整页或全站。
- 无法证明章节映射安全、比较结果可能被截断、基线提交不可达或翻译破坏结构时，立即失败并保留原文件。不得静默全量扫描、全篇重译或回退。
- 只有无关代码变化时，只更新 `content/source.json` 的已核对提交和日期；不得调用模型、改写正文、修改导航或生成变更日志。
- 保持链接目标、图片路径、HTML、围栏代码块、行内代码、命令、API、类型、变量、布尔值和版本示例不变。构建阶段必须根据目标英文标题为静态网页补齐官方锚点，保证原文快捷链接可用。
- 图片按上游 `docs/images/` 下的相对路径同步到 `public/docs-images/`；只写入比较结果列出的图片。
- 中文文档变更日志继续存储在 `content/changelog.json`，由静态网页“中文文档变更日志”栏目展示。不得创建 Markdown 日志。
- 未经用户明确要求，不提交、不推送、不修改触发方式，也不发布任务范围外的内容。

## 完成报告

报告基线与目标版本/提交、实际处理的页面与章节数量、图片和导航变化、日志条目，以及每项验证结果。若停止，明确指出触发的契约条件和未写入任何内容的证据。
