# Pi 中文文档增量更新契约

本文规定 `scripts/sync-upstream.mjs` 和调用它的代理必须遵守的输入、处理、写入及停止条件。目标是让每次更新可追溯、局部、可验证，不依赖代理自行判断是否全量扫描。

## 1. 权威输入

- 上游仓库固定为 `earendil-works/pi`。
- 文档根目录固定为 `packages/coding-agent/docs`。
- 本地基线固定读取 `content/source.json` 的 `commit` 和 `version`。
- 默认目标是上游 `main` 当次解析得到的完整提交 SHA；提供 `--target` 时，先解析为完整 SHA，再固定使用它。
- 比较范围固定为 `<基线提交>...<目标提交>`。

禁止用本地文件时间、模型知识、上次任务输出或浮动 `raw/main` 代替提交差异。

## 2. 比较结果

只处理以下路径：

| 上游路径 | 本地目标 | 行为 |
|---|---|---|
| `packages/coding-agent/package.json` | `content/source.json` | 读取版本，不复制文件 |
| `packages/coding-agent/docs/docs.json` | `content/navigation.json` | 增量重建导航 |
| `packages/coding-agent/docs/*.md` | `content/docs/*.md`、`content/source-structure.json` | 按状态和章节处理 |
| `packages/coding-agent/docs/images/**` | `public/docs-images/**` | 按字节新增、替换、删除或重命名 |

其他路径只计入上游比较信息，不得触发文档翻译或本地内容写入。

满足任一条件必须在写入前失败：

- GitHub Compare API 无法确认基线与目标的关系。
- 比较状态不是 `ahead` 或 `identical`。
- 返回的变更文件列表可能达到 GitHub Compare API 的 300 文件上限。
- 任一相关条目缺少文件名、状态，或重命名条目缺少 `previous_filename`。
- 基线提交、目标提交或所需文件无法按 SHA 读取。

不允许用遍历全部文档、下载整个仓库、改用 `main` 原始地址或整站重译作为回退。

## 3. Markdown 章节合并

### 3.1 章节边界

- 在围栏代码块之外，以一级至四级 ATX 标题作为章节起点。
- 第一个标题前的内容是前言章节。
- 章节原文包含标题行及其后正文，直到下一个一级至四级标题或文件末尾。
- 使用规范化换行后的章节内容计算 SHA-256；只忽略章节边界处的尾随空白，以便末章移动后仍能复用，章节正文中的空格、链接和代码差异均不得忽略。

### 3.2 复用条件

修改页面必须同时读取：基线提交中的旧英文、目标提交中的新英文、现有中文页面。

1. 旧英文与现有中文的章节数量和标题层级必须逐项一致，否则停止。
2. 新英文每个章节按完整内容哈希匹配尚未使用的旧英文章节。
3. 匹配成功时，直接复用同索引的现有中文章节，内容不得经过格式化、翻译或重写。
4. 同一内容移动到新位置仍按哈希复用。
5. 未匹配的新章节才进入翻译。

删除的章节不写入新中文文档。新增页面允许整页翻译；删除页面直接删除本地文件；重命名页面使用旧路径中文作为合并输入并写入新路径。

若旧英文存在内容完全相同但对应中文不同的重复章节，或无法唯一映射旧英文与中文，停止处理该页面，不得整页重译。

### 3.3 翻译约束

- 忠实逐句翻译自然语言，不摘要、不删节、不补充。
- 保持 Markdown/HTML 结构、标题级别、链接目标、图片路径、围栏代码块、行内代码不变。
- API、类型、变量、命令、布尔值、版本号和示例值保持英文。
- 每个新翻译章节执行严格结构与技术字面量校验；组装后的整页再校验章节映射。
- 允许对同一变化章节重试一次；第二次仍失败则整个同步失败。

### 3.4 标题锚点与快捷链接

- 现有中文源码可能含有历史显式锚点，也可能完全不含；不得为了统一格式扫描或重写未变化页面。
- 未变化章节连同其历史锚点逐字节复用。现有基线只校验章节数量、标题层级和前置元数据，以容纳既有翻译排布。
- `content/source-structure.json` 使用与网页渲染器相同的标题 slug 规则，保留下划线并为重复英文标题追加稳定后缀。
- 构建阶段由 `scripts/generate-docs.mjs` 按目标英文标题给静态网页补齐缺少的官方锚点；最终链接测试必须证明页内快捷链接、上一篇和下一篇导航有效。

## 4. 导航与图片

- `docs.json` 未变化时，`content/navigation.json` 必须保持原样。
- `docs.json` 变化时，同时读取基线和目标配置；英文标题未变化且路径未变化时复用现有中文，新增或英文标题变化时只翻译对应短标题。
- 始终在导航末尾保留唯一的“项目 / 中文文档变更日志”项，其路径为 `changelog`。
- 图片内容按字节写入，不得转码或重新压缩。
- 图片重命名先读取目标提交的新内容，成功完成全部校验后再删除旧路径。

## 5. 事务写入与状态

同步分为“读取和计算”与“候选应用及验收”两阶段。网络、解析、翻译或预写入校验错误发生时，不得写入任何项目文件。

全部候选内容计算完成后，脚本先为所有受影响文件和 `app/generated-docs.json` 建立内存快照，再按以下顺序应用：

1. 写入新增、修改和重命名后的中文文档及图片。
2. 删除比较结果明确标记为删除或已重命名的旧文件。
3. 更新 `content/navigation.json`、`content/source-structure.json` 和 `content/changelog.json`。
4. 更新 `content/source.json` 候选基线。
5. 依次执行结构检查、增量测试、`npm run build:pages`、`npm run build` 和完整测试。

任一应用步骤抛出可捕获异常，或验收子进程以非零状态退出时，必须恢复快照并删除本次新建文件；因此这类失败后的 `content/source.json`、日志、正文、图片和生成数据仍保持运行前状态。进程被强制终止、宿主崩溃等无法执行 `catch` 的情况不属于该回滚保证；在持久工作区再次运行前必须先检查工作树。

同一目标提交已经存在于 `content/changelog.json` 时不得重复添加日志。基线与目标相同且没有相关变化时不得改写内容或基线，但仍执行验收以产出部署构建目录。

若基线与目标不同但比较范围只有无关代码变化，只候选更新 `content/source.json` 的 `commit` 和 `syncedAt` 作为已核对检查点；版本保持不变，不调用模型，不改写正文、导航、结构或图片，也不生成变更日志。该检查点仍须通过同一事务验收后才能保留。

## 6. 中文文档变更日志

沿用当前 JSON 数组及字段：`date`、`version`、`commit`、`summary`、`added`、`changed`、`removed`。

- `summary` 写明旧版本到新版本及短提交范围。
- `added` 记录新增页面和图片。
- `changed` 记录修改页面、导航、图片和仅版本变化。
- `removed` 记录删除页面和图片。
- 页面条目优先使用中文导航标题；没有中文标题时使用相对路径。
- 只记录本次比较范围中的事实，不根据提交标题推测文档变化。

## 7. 验收

同步命令内部至少依次完成：

1. `node scripts/validate-doc-structure.mjs`。
2. 增量引擎离线测试，覆盖无变化、只有无关变化、单章节变化、章节移动、增删改名、导航/图片、比较上限、技术字面量、日志去重和失败回滚。
3. `npm run build:pages`。
4. `npm run build`。
5. `node --test "tests/*.test.mjs"`。

最终还必须确认：目标英文与中文标题层级一致，代码围栏闭合；链接目标、图片路径、代码和技术字面量未被翻译破坏；计划外中文章节逐字节不变；静态网页的官方英文标题锚点、页内链接、上一篇和下一篇链接有效。

任一可捕获检查失败时，事务必须回滚，不得把 `content/source.json` 的候选基线视为已更新。
