# 云端增量同步 Pi 中文文档

完全通过已连接的 GitHub 和 Outlook 工具执行。不得依赖本地仓库、Shell、GitHub Actions、未暴露的访问令牌或持久 worktree。

执行前读取 [云端同步契约](references/cloud-contract.md)。冲突时采用更严格的停止条件。

## 固定范围

- 中文仓库：`FB208/pi-coding-agent-cn`，目标分支：`main`。
- 上游仓库：`earendil-works/pi`，目标分支：`main`。
- 唯一原文目录：`packages/coding-agent/docs/`。
- 基线：中文仓库 `content/source.json` 中保存的完整上游 commit SHA。
- 结果通知：使用已连接的 Outlook 邮箱发送到任务提示指定的收件人。

不要重新建立历史基线，不要创建 `DOCS_CHANGELOG.md`。沿用仓库当前的 `content/source.json`、`content/source-structure.json`、`content/navigation.json` 和 `content/changelog.json`。

## 工作流

1. 读取两个仓库及 `main` 的当前状态。读取本 Skill、仓库内原始 Skill 和其更新契约，以仓库内更严格的新规则补充本流程。
2. 固定解析上游 `main` 和中文仓库 `main` 的完整 SHA。比较 `content/source.json.commit...上游目标 SHA`，仅保留契约允许的文件。目标 SHA 在整次运行中不得漂移。
3. 若比较为 `identical`，或没有任何文档相关变化且基线已是目标 SHA，不写入 GitHub；完成只读校验后发送“无变化”邮件。
4. 对相关文件按状态处理：
   - Markdown 修改或重命名：读取旧英文、新英文和现有中文，按围栏外 H1–H4 标题分块；按规范化块哈希复用未变化中文，只翻译无法匹配的新块。
   - Markdown 新增：只翻译新增页面。删除：删除对应中文页面。
   - 导航、结构、图片、版本和变更日志：只处理比较结果明确列出的内容，保持仓库既有格式和映射。
5. 在内存中完成候选内容和必要验证。验证通过前不得创建分支或写文件。
6. 从当次读取到的中文仓库 `main` SHA 创建唯一日期化分支，例如 `docs-sync/2026-07-19-<上游短SHA>`。把所有候选文件组成一个提交；不要逐文件直接提交到 `main`。
7. 创建非草稿 PR，正文列出固定上游 commit 范围、变更页面、验证结果和自动合并说明。确认 PR base 是 `main`、head SHA 与刚创建提交一致。
8. 立即自动合并该 PR 到 `main`，优先使用 squash。不要等待人工复核，也不要把非强制检查作为阻塞条件。若 GitHub 因必需检查暂未完成而拒绝立即合并，且仓库支持自动合并，则启用自动合并；若分支保护、必需检查失败、冲突、权限或并发变化最终阻止合并，禁止 force，保留未合并 PR 并按失败处理。
9. 重新读取 `main`，确认合并结果包含预期目标 SHA、元数据和变更日志且没有计划外内容。
10. 无论成功、无变化或失败，发送纯文本结果邮件；邮件发送失败时在任务结果中明确报告。

## 强制停止条件

- Compare 关系不是 `ahead` 或 `identical`，结果可能触及 300 文件上限，或条目缺少状态/重命名来源。
- 基线、目标或必要文件不能按固定 SHA 读取。
- 旧英文、现有中文及保存结构无法唯一映射，重复块对应不同译文，或需要整页/整站重译才能继续。
- Markdown 标题层级、代码围栏、链接目标、图片路径、行内代码、技术字面量或 JSON 结构验证失败。
- 计划外中文正文发生变化，候选 diff 不完整，或无法原子地构造单一提交。
- `main` 在写入期间产生无法自动解决的冲突、GitHub 拒绝合并或无法确认合并结果。

停止时不得更新 `content/source.json`，不得合并，不得直接写 `main`。如果已经创建 PR，保留它并在标题或正文中明确标记自动同步失败，不执行合并。

## 翻译与复用

- 忠实逐句翻译自然语言；不摘要、不删节、不补充。
- 完全相同或仅移动的块逐字复用现有中文，不得重新格式化。
- 保持 Markdown/HTML、代码块、行内代码、命令、类型、变量、API、URL、锚点、版本号和示例值准确。
- 新块最多重试翻译一次；仍无法通过结构和技术字面量检查则停止整个同步。

## 云端验证

云端 Worker 无本地项目目录时，不声称执行 `npm`、Node 或构建测试。至少完成：比较完整性、文件白名单、JSON 解析、Markdown 分块与围栏、技术字面量、链接/图片路径、计划外差异、日志去重、基线/目标一致性和合并后回读验证。只有 GitHub 标记为必需且明确失败的仓库检查才阻止合并；其他检查不要求人工等待。

## 邮件格式

- 成功：`[Pi 文档同步] 成功 - YYYY-MM-DD`，包含上游范围、页面、合并 commit/PR 链接和验证结果。
- 无变化：`[Pi 文档同步] 无变化 - YYYY-MM-DD`，包含已检查的目标 SHA 和时间。
- 失败：`[Pi 文档同步] 失败 - YYYY-MM-DD`，包含失败阶段、准确原因、是否创建分支/PR、`main` 是否保持不变及处理建议。

最终任务报告必须与邮件状态一致，不得把“创建 PR 但未合并”报告为成功。
